## 问题 1：定价（最优先）

### 权威公式（用户 `底层信息/定价公式.xlsx`，与乐羽/现后端逐字一致）
| 列 | 公式 |
|---|---|
| 价格 | `总成本 / 百分比`（百分比范围 **0.28–0.78**） |
| 利润 | `价格 − 总成本` |
| 活动扣点 | `价格 × 0.07` |
| 二级利润 | `利润 − 活动扣点` |
| **利润率** | `二级利润 / 价格` = `1 − 百分比 − 0.07` |
| 毛保本投产 | `1 / 利润率` |
| 拼单价 | `价格 + 20` |

样例：成本 11.3，百分比 0.75 → 价格 15.07、利润率 **0.18**。**公式本身没错，后端 doCalculate 已与它一致。**

### 现象（截图自证）
成本 7.80 → 售价 42.29 → 利润率显示 81.6%。

### 根因（全在前端口径，非公式）
1. **滑块口径拧反**：滑块拖的是"百分比"（成本占价格=公式分母），不是利润率。二者关系 `利润率 = 1 − 百分比 − 0.07`。用户要"拖 33% 则该列全 33%"，故滑块应直接拖**目标利润率 m**，后端反解 `百分比 = 0.93 − m`（0.07 是活动扣点），利润率列恒等于 m。
2. **绿色列 `marginPct` 算错两处**：现为 `(售价−成本)/售价`——① 漏减 7% 活动扣点（正确是"二级利润/价格"=公式列，即后端已算好的 `marginRate`）；② 用的售价是 pinPrice(价格+20) 而非"价格"。截图 81.6% = (42.29−7.80)/42.29，两错叠加；正解 58%(=1−0.35−0.07)。
3. **平台价 vs 导出差 20**：前端回填 `groupPrice = pinPrice`(价格+20=42.29)，导出表"价格(活动价)"列是**价格**(22.29) → 差 20。

### 方案（复用乐羽公式，只改前端口径 + 后端加 profitRate 入参）
1. **滑块改「利润率」**：v-model 直接是利润率 m（范围如 15%~65%，对应百分比 0.28~0.78），label 改"利润率"。
2. **后端 `doCalculate` 支持 profitRate 入参**：若传 profitRate 则 `百分比 = 0.93 − profitRate`（clamp 到 0.28~0.78），其余公式不动。marginRate 输出恒 = profitRate（自洽）。仍兼容老的 costRatio 入参。
3. **前端绿色列直接用后端返回的 `marginRate`**（= 二级利润/价格），不再前端自算 `(p-c)/p`；与滑块 m 天然一致。
4. **回填价**：见待拍板 A——平台/导出统一取"价格"还是"拼单价(+20)"。

> ⚠️ 待你拍板 A：平台发品价 = "价格" 还是 "拼单价(价格+20)"？这是拼多多拼单玩法的业务决定。无论选哪个，**平台价与导出表必须同源同字段**（现在的 bug 就是两边取了不同字段）。

## 问题 2：详情图 / SKU 图未生成（**合并时没接全，不只是超时**）

用户原话："这些工作不应该是合并时就做好的吗？" —— 核对后确认：**是合并缺陷**。乐羽 537 行 `generateSkuImage` 整段迁进了云端 `lyimage/`（行数一致），但 `FlowController.step2` 调用它时**把关键入参填成了空占位**，成熟的防比价/贴图逻辑全空转。

### 缺陷 A：SKU 生图三个入参在合并时被填空（核心）
对比两处调用（签名同）：

乐羽 `ListingController.genSkuImages`（可用）：
```
generateSkuImage(ref, name, comp, type, batch, seq,
  bagImagePath, whiteImgPath, accImagePaths,  waterImagePath,
  bgStyle,          // 整批 analyze-bg 一次的共享背景
  itemCode, accParts, templateId);   // 防比价模板
```
GOFU `FlowController.step2`（坏）：
```
generateSkuImage(refMain, name, spec2, type, batch, i+1,
  "", skuWhite, List.of(), "",   // ← bag/accImagePaths/water 全空
  "",                            // ← bgStyleOverride 空
  itemCode, accParts, "");       // ← templateId 空
```
后果，逐条：
1. **`accImagePaths = List.of()`** → `ImageGenService` 里 `candFiles` 只从 accImagePaths 构造（行189-196），空 → `accFiles` 空 → `compositeShowerLeft` 没有任何配件卡可贴。**贴图合成（确定性防比价核心）产出空**。accParts 拆组合码逻辑照跑，但匹配的是 0 张图。→ 直接对应用户「SKU 没搭配件」。
2. **`templateId = ""`** → `tpl=null`，退回默认 sticker，**防比价模板（ai 图生图/基准图复用）根本没触发**。用户记得的"防比价机制"没启用。
3. **`bgStyleOverride = ""`** → 命中 CLAUDE.md 雷区原文："整批只 /analyze-bg 分析一次…并发下别让每个 SKU 各自重新分析"。现每个 SKU 各自重新分析背景 → 背景不一致 + 每 SKU 多一次多模态调用（**这也是拖到超时的元凶之一**）。

### 缺陷 B：叠加超时（红字自证）
`云端转发失败: Read timed out`。云网关 readTimeout=300s，step2 一次性同步生成剩余主图+详情+全部SKU，再加缺陷 A 每 SKU 重复分析背景 → 累计超 300s → 整个 HTTP 丢失，已生成的也没回写 → "只有一张主图"(step1首图)。

### 缺陷 A′：配件白底图**根本没被拉取**（比 A 更上游）
核对前端 `autoFetchWhite`：`codes = this.mainCodes`——**只拉主件白底图**，配件（软管/底座/滤芯）的白底图从没从快麦下载过。所以即便把 accImagePaths 接上，也没有配件图文件可传。
乐羽的模型：主件+配件白底图都在**同一个文件夹**，`siFindAccessoryImages` 扫这个文件夹、排除袋子/水质/已用作主件的图 → 得到配件图列表。GOFU 走快麦下载，则需**把配件编码也一并 fetch-white-images 到同一 erp-white 目录**。

### 修复方案
1. **拉取配件白底图**（新增，最上游）：`autoFetchWhite` 的 codes 除主件外，加上已搭配的配件编码（accParts 的 code / 已选 accessory·batch 的 itemCode），一起 fetch 到同一 erp-white 目录。matched 落 `entry.whites`，或单独记 `entry.accWhites`。
2. **接全 SKU 生图入参**（复用乐羽 `genSkuImages` 的组织方式）：
   - `accImagePaths`：erp-white 目录里的配件图（排除主件色图/袋/水质）传进去。
   - `bgStyle`：step2 开头对首张主图 `analyzeBackgroundStyleOnce` **只调一次**，分发给所有 SKU（对齐雷区规则）。
   - `templateId`：默认 `sticker-leftcard`（贴图合成，确定性），或按需选 ai 模板。注意空串已是 sticker 模式，真正阻塞是 accImagePaths+bgStyle。
2. **详情图 16:9**（用户已拍板）：现 prompt 是 9:16 竖版，改 16:9 横版、尺寸随之改，复用羽刃对应话术。
3. **SKU 图参考主图背景**（用户："参考主图的背景"）：`refMain` 传首张主图已对，关键是 bgStyle 从首张主图分析并共享（见上）——这正是乐羽"防比价背景一致"的设计。维持乐羽贴图/基准图机制。
4. **超时**：接全入参 + bg 只分析一次后单次耗时会降；SKU 多时仍需拆批或异步轮询兜底（羽刃已有 task 轮询 + 前端 `pollTask`）。→ **待拍板 D**。

## 问题 4：营销词不统一 → 虚假宣传

### 根因
`generateSkuPlans` prompt 让每个主件 specName **各自**带营销卖点（"月光银-过滤净水"/"雅黑色过滤按摩莲蓬头"），LLM 给不同颜色编了不同功能词，出现某色写"按摩"但该花洒无此功能。

### 方案（改 prompt 约束）
在命名规范里加硬约束：**同一批主件的营销卖点词必须统一**，只有颜色/款式名可不同，功能卖点（增压/过滤/亲肤等）取该系列共有功能，**禁止给个别颜色臆造独有功能词**。属 prompt 工程，改 `LyTextService` 一处。
