# M10 · 07.08 编排重构方案（已确认，待批准动手）

## 用户确认的决策
1. **两段式流程**：先出布局+主图(并行,快) → 用户预览选定方案 → 再点"生成图片"只对选定方案生 SKU图+详情图。避免白生没选中方案的图（解决"三套以上太慢"）。
2. **详情图配对**：详情张数=主图张数，第 i 张详情由第 i 张主图转 9:16（内容一一对应，不再独立循环）。
3. **布局与主图并行**：布局靠 ERP选品+规则库，主图靠白底图，互不依赖，后端并行跑。
4. **问题3 架类型号命名**：prompt 按品类区分，架类禁止臆造层数档位。
5. 问题1（架类/花洒没出图）：**独立于本重构**，等用户发 cloud.log 定位（疑似该测试机网络连不上生图接口）。

## 依赖关系（重构依据）
```
布局(ERP/规则,靠千问) ─┐
                       ├── 并行,互不依赖
主图(白底图,靠gpt-image)┘
                              ↓ 选定方案后
                         ┌─ SKU图(参考首张主图 + SKU白底 + 配件白底)
                         └─ 详情图(第i张主图 → 第i张9:16详情)
```

## 改动方案

### A. 后端编排：三阶段（FlowController）
- **step1（布局+全部主图，并行）**：
  - 布局(generateSkuPlans) 和 主图(生成 mainCount 张，不只首图) 用 `imageGen.getExecutor()` 两个 future 并行，`CompletableFuture.allOf` 汇合。
  - 主图现在 step1 出**全部** mainCount 张（原来只出首图、剩余丢 step2）。因为主图不依赖方案选择，越早出越好、还能并行。
  - 返回 planCount + 主图。仍异步(返回 taskId, 前端轮询)——主图多张耗时，走现有 GenerationTask 机制。
- **step2（选定方案后：SKU图 + 配对详情图）**：
  - 入参加 `planIndex`（用户选的方案）。只对该方案生 SKU 图。
  - 详情图：`for i in 主图: detail[i] = 主图[i] 转 9:16`（配对，数量=主图数）。
  - SKU 图逻辑不变（参考首张主图+配件白底+accParts+防比价模板）。

### B. 前端两段式（workbench.html）
- 「🚀 一键生成」拆成语义两步（或保留一键但中间停）：
  - 点生成 → 调 step1(布局+主图) → 轮询完 → 预览区展示布局各方案 tab + 主图。
  - 用户在方案 tab 选定 → 「生成该方案的 SKU 图和详情图」按钮 → 调 step2(带 planIndex)。
  - 设置「方案数」保留(默认3)；planCount 传 step1。
- 详情图张数选择器移除（改成"跟随主图张数"，不再独立配置）。

### C. 问题3：架类型号命名（LyTextService.generateSkuPlans prompt）
- 命名规范按品类分支：
  - **花洒**：保留现有(增压/软管/滤芯/喷头 卖点词)。
  - **架类/通用**：型号名只描述**配件组合**(如"+挂钩""+隔板""+沥水盘")，**禁止臆造层数/规格档位**（主件的层数/尺寸等固有属性已在主件名，型号不得再编"单层款/三层款"这类与主件冲突的档位词）。卖点词改通用(稳固承重/免打孔/加厚防锈/大容量)。
- 实现：prompt 里判断 category 含"花洒/淋浴"用花洒规范，否则用通用规范段。参考羽刃按品类切 prompt 的思路(catalog/*.txt)。
