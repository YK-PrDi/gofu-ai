# GOFU-AI 一键生成 → 上新 全链路重构分析

> 目的：测试反馈 7 个 bug + 4 条建议，根因调查 + 复用乐羽/羽刃成熟代码的重构方案。
> **本文档供确认，确认后再动手改代码。** 调查基于三个只读探查，路径均精确到行号。

---

## 一、一句话结论

上一批（M9修复）把工作台的**调用链接通了，但业务逻辑是"框架+静默降级"**：
- 生图链每一步失败都不报错、只 log、返回计数 0 继续走 → 一处坏就大面积空（bug1）
- 前端定价只算**当前选中的那一套**方案、且成本**漏了主件** → bug2/3/4 连锁
- 上新**默认诊断模式**(dryRun)不真提交，且"任务完成"≠"发品成功" → bug6
- 标题是链路末端 best-effort，前面一错就轮不到它，空标题又不兜底 → bug7

**乐羽的 `lyAutoRun` 和羽刃的多图/生图逻辑都是成熟且测试过的**，重构方向 = 用它们替换掉当前的框架实现，而不是继续打补丁。

## 二、7 个 bug 的根因（调查实证）

### bug1：只有一张主图，详情图/SKU图全无
- **链路**：step1 出首图(1张) → step2 出剩余主图(默认2张)+详情图+SKU图。
- **单点致命闸门**：`FlowController.java:144` step2 开头 `localizeWhite(whiteImages.get(0))` 失败即直接 return，**主图/详情/SKU 全灭，只剩 step1 那 1 张**。COS 未启用时首图存的是云端本地临时路径，跨 step/进程易失效，放大此问题。
- **全链 best-effort**：`GptImageAgent` 生图失败返回 false，`FlowController` 一律跳过不 add、不重试；HTTP 恒 200，前端只读计数不校验（`workbench.html:707`）。密钥/配额/白底任一问题→静默坍缩成 1 张。
- **SKU图额外依赖** plans 非空（step1 的 `generateSkuPlans` 靠千问 key，失败则 plans 为空 → SKU 图整段跳过）。

### bug2：只有第一套方案有定价/利润
- `fillCostAndPrice()`（`workbench.html:737`）只处理 `this.currentItems`（**当前选中方案**），从不遍历 `this.plans`。
- runAll 里调用时 `selectedPlan` 恒为 0 → 只有 plan[0] 的 items 拿到价；`selectPlan` 切换方案（`:821`）**不重算**。其余方案 groupPrice 恒为建项时的 0。

### bug3：方案成本不对（偏低）
- `fillCostAndPrice` 组装成本请求时 components **只放 accParts（配件），漏了主件**（`workbench.html:742-747`）。
- 正确做法在 `preview.html:435,458-466`：先 `it.itemCode.split('+')[0]` 取主件码作 components 首项，再加配件。乐羽 `leyu/app.js:976` 同样主件永远是第一项。
- 结果每个 SKU 成本 = 只有配件料+运费，**漏了主件（喷头本体）采购价**。

### bug4：上新定价输成 0
- bug2/3 的下游后果：前端只给 plan[0] 算了价、且没保证落库 → context 里多数方案 groupPrice=0。
- `ContextToListingMapper.java:106` 无脑 `toCents(it.getGroupPrice())`，对 0 无校验无兜底 → 上新输 0。

### bug5：SKU名/营销词需优化
- 当前 SKU 名来自云端 `LyTextService.generateSkuPlans`，营销词规范不如乐羽 `ListingService.java:649-656`（颜色+营销卖点如"【雅黑色】过滤按摩增压"、型号"过滤喷头+5个滤芯【半年装】"，卖点词库6-15字）。

### bug6：上新点确认后拼多多后台没链接
- **根因A（最直接）**：`workbench.html:456` `listingDryRun:true` **默认勾诊断模式**。dryRun 时脚本(`pdd_listing.js:1698-1704`)只截图就 return，从不提交，任务却报 done → 前端显示绿色成功，但后台没商品。
- **根因B**：`TaskService.submit:74` 用"进程正常退出"定义 done，与脚本内 `DONE/ERROR` 语义脱钩。脚本 `error()` 只打印 ERROR 行不让 Java 任务变 error → 前端 `pollTask:915` 只看 status，**"任务完成"≠"发品成功"**。
- **根因C**：提交前多步 best-effort，必填项（标题/分类/价格/SKU图）缺失→拼多多校验拦截→停编辑页无链接。

### bug7：没生成"链接名"(标题)无法预览
- 标题生成 `genTitle()` 在 runAll **末端**（`workbench.html:704`）。前面 step1/step2 抛错则 catch 跳出，genTitle 根本没执行 → 标题空。
- 消费侧 `ContextToListingMapper.java:77` 取空标题不兜底；`pdd_listing.js:677` 标题空则跳过填写。拼多多标题必填 → 无法预览/提交。

## 四、重构方案（复用乐羽/羽刃成熟代码）

### R1. 定价链修复（bug2/3/4）— 改 `workbench.html:fillCostAndPrice`
参照 `preview.html:fetchCosts()`（成熟实现）重写：
- **遍历所有 plans**（不只 currentItems），每套方案都算价。
- components **先加主件码** `it.itemCode.split('+')[0]` 再加 accParts（修 bug3 漏主件）。
- 算完**立即 saveContext 落库**，切换方案不用重算（修 bug2）。
- `ContextToListingMapper.java:106` 加 groupPrice<=0 告警/拒绝上新（修 bug4 兜底）。

### R2. 多图数量可选（默认6）（bug1 + 建议3）— 复用羽刃
- 前端加"主图张数/详情图张数"选择器，默认 6（复用羽刃 `#customCount` index.html:417 机制，默认改6）。
- `FlowController` step2 主图循环数量由前端传入（现在写死 extraMainCount=2）。
- **去掉单点闸门**：step2 白底本地化失败不应全灭，改为每张独立 best-effort + 失败重试（羽刃 `custom_generate` 有串行建基调+并行重试）。

### R3. SKU 图生图（bug1 SKU图 + 建议4"sku生图没做"）— 复用乐羽 `generateSkuImage`
- 当前 SKU 图是 `FlowController` 里简单 generateImageMulti，没有乐羽的 sticker 贴图（配件卡确定性合成）/ai基准图复用两套成熟逻辑。
- 复用乐羽 `ImageGenService.generateSkuImage`（sticker+ai双模式）+ `compositeShowerLeft` 配件卡合成。

### R4. 上新成功语义修复（bug6）
- 前端默认 dryRun 改为 false（或明确二次确认）。
- `TaskService`/脚本：把脚本的 `DONE:success/ERROR` 语义传导到任务状态，前端据此判真实成败，不再"进程退出=成功"。

### R5. 标题前置+兜底（bug7）
- 标题生成移到 step1 之后（不放末端），失败兜底一个基础标题（品牌+品类+卖点），保证非空。
- 消费侧空标题拒绝上新并提示。

### R6. SKU名/营销词优化（bug5）— 复用乐羽 prompt
- 复用乐羽 `ListingService.java:649-656` 的营销词规范到云端 `generateSkuPlans` prompt。

### R7. 选品交互改"打开即显示"（建议1）
- 当前需搜索才有内容；乐羽 `openErpModal`(app.js:2064) 打开即全量加载+常用置顶。复用之。默认选中 role=main。

### R8. 白底图预览（建议2）
- 白底图检测/索取后加缩略图预览。

## 五、已定决策（用户确认）

1. **分两批**：
   - **第1批（必修bug，改动小风险低）**：R1定价链 + R4上新成功语义 + R5标题前置兜底 + R7选品打开即显示 + R8白底图预览。
   - **第2批（生图重构，改动大）**：R2多图数量可选(默认6) + R3 SKU生图全量迁乐羽 + R6营销词优化。
2. **SKU生图全量迁**：乐羽 `generateSkuImage`（sticker贴图合成 compositeShowerLeft + 防比价模板 + 基准图复用）全量迁入，不做简化版。
3. **架构分工确认**：生图逻辑落云端 gofu-server-cloud，依赖的 compositor/模板/基准图一并迁。
