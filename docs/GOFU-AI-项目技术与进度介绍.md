# GOFU-AI 电商全流程自动化中枢 — 技术与进度介绍

---

## 1. 项目概述

**GOFU-AI 是一套电商全业务流程自动化中枢**，把电商上架前最耗人力的两件事——**商品视觉生产（生图）** 与 **商品上架（上新）**——用 AI 与自动化串成一条流水线。

一句话定位：**选好品类、导入产品白底图，系统自动完成"出图 → 定价 → 上架"，把人从重复劳动里解放出来。**

它解决的核心痛点：

| 痛点 | 传统做法 | GOFU-AI |
|------|----------|---------|
| 商品图制作慢、成本高 | 摄影棚拍摄 / 美工逐张 P 图 | AI 按产品白底图批量生成主图、详情图、SKU 图 |
| 同款商品易被比价 | 图片雷同，平台一搜就比价压价 | 防比价生图：每款风格、构图、卖点差异化 |
| 生图与上架数据割裂 | 做完图存本地，手动复制链接去上架 | 统一"商品上下文"打通，数据一次录入全程复用 |
| 上架操作繁琐、易触风控 | 人工逐个填规格、传图、定价 | 自动驱动，真实环境操作 + 关键环节人工介入 |

---

## 2. 核心价值

**① 全流程自动化闭环**
从选品类、导入白底图开始，系统自动：AI 布局规划 → 生成多张营销主图 → 生成详情图 → 按 SKU 组合生成配件展示图 → 计算成本与阶梯定价 → 生成标题 → 一键上架到电商平台。人工只在"选定方案"和"平台验证码/滑块"两个环节介入。

**② 防比价生图**
电商同款商品最怕被平台比价压价。系统为每款商品生成**风格、构图、视角、画面卖点文案都差异化**的图片，从视觉上与同行拉开区分，降低被比价的概率。

**③ 视觉流反哺结构流（差异化设计）**
这是本项目区别于普通"AI 生图工具"和"上架工具"的关键：生图阶段 AI 提取的**商品卖点**，会自动反哺到上架阶段的 **SKU 规划**（围绕卖点策划商品型号与组合）。生图和上架不再是两个孤立工具，而是**一条互相增强的数据链**。

## 3. 能力总览

当前系统已具备的能力：

- **AI 营销主图生成**：读取产品白底图，生成高级感场景营销图，保持产品主体真实一致（防虚假宣传），支持多种画面风格随机切换。
- **SKU 配件展示图**：按"主件 + 配件"组合（如花洒 + 底座 + 软管 + 滤芯）生成带配件说明的展示图，配件部分用真实白底图确定性合成，100% 还原。
- **详情图生成**：9:16 竖版详情图，与主图保持视觉连续。
- **智能配件搭配 + 阶梯定价**：按规则库自动补配件、算成本、出阶梯价、生成标题。
- **一键上架**：驱动真实浏览器环境完成平台上架，触发验证码/滑块时暂停并提醒人工介入，无缝续跑。
- **多品类支持**：花洒品类已完整闭环，正在扩展"厨房架类品"（刀架、挂钩、锅盖架、沥水架、置物架等）。

## 4. 技术架构

### 4.1 总体架构：Monorepo + 云/本地职责分离

系统采用单仓多模块（Monorepo）结构，分三个模块，**云端与本地物理隔离、职责不可混**：

```
┌───────────────────────────────────────────────────────────────┐
│                        GOFU-AI Monorepo                       │
├───────────────┬───────────────────────┬───────────────────────┤
│  gofu-shared  │   gofu-server-cloud   │   gofu-client-local   │
│  （共享契约）   │  （云端·视觉流/算力）     │  （本地·结构流/反风控）   │
│               │       端口 5020        │       端口 5021        │
├───────────────┼───────────────────────┼───────────────────────┤
│ ProductContext│ · AI 生图（多模型）      │ · 上新流程调度          │
│ DTO / 枚举     │ · 商品上下文权威存储      │ · 反风控（真实环境）     │
│ 无业务逻辑      │ · Canvas 图层合成       │ · Canvas 拼装前端      │
│               │ · 部署腾讯云            │ · Electron 桌面端      │
└───────────────┴───────────────────────┴───────────────────────┘
        跨模块只走 gofu-shared 契约，云/本地互不 import 对方业务类
```

- **gofu-server-cloud（云端）**：AI 生图吃算力、需集中计费，放云端（腾讯云）。是所有生图能力的唯一收口，也是商品上下文的权威存储。
- **gofu-client-local（本地）**：上架依赖真实的本地网络环境与浏览器指纹做反风控，必须在本地跑，以 Electron 桌面应用形态交付。
- **gofu-shared（契约）**：只放跨端传输的数据契约（商品上下文、DTO），不含任何业务逻辑。云端和本地在物理上都无法 import 对方的业务代码——**架构隔离本身就是护栏**。

### 4.2 商品全局上下文（ProductContext）——系统脊柱

整个系统围绕一个统一的"商品全局上下文"对象运转。它像一根脊柱，把散落各环节的数据挂在一起：

- 生图产物（卖点、主图、详情图、SKU 图）
- 上新产物（SKU 组合、成本、定价、标题）
- 人工锁定的字段

**云端是这个上下文的唯一权威源**，本地按需拉取、人工微调后回写。这样"先做图 → 存本地 → 手动传链接"的割裂被彻底打通，数据一次录入、全流程复用。

### 4.3 技术栈

| 层 | 技术 |
|----|------|
| 后端 | Java 17 + Spring Boot 3.3.x |
| 桌面端 | Electron（本地客户端，内嵌 JVM + 前端） |
| 浏览器自动化 | Playwright（驱动真实 Chromium 做上架与反风控） |
| 前端 | Vue3（CDN 免构建挂载，用于交互最重的预览页） |
| AI 生图 | 多模型 Agent（Gemini / GPT-Image / 通义千问 / 万相 / 混元），统一接口、可降级 |
| 存储 | SQLite（上下文持久化）+ 腾讯云 COS（图片对象存储，永久 key + 按需换签名 URL） |

**AI 生图采用多模型可降级设计**：统一的生图 Agent 接口下挂多个模型，主模型不可用时自动降级到备用模型，保证出图链路的稳定性。

## 5. 关键设计决策

项目以"架构决策记录（ADR）"方式沉淀关键取舍，以下摘取几条对外有代表性的：

- **生图能力唯一收口云端**：所有 AI 生图/重绘只在云端，本地要图就通过网关 HTTP 调云端。避免两边重复实现、算力集中计费。
- **物理隔离即护栏**：云端与本地模块之间没有代码依赖，物理上无法互相调用对方业务类。跨端只能走契约层，从根上杜绝职责混乱。
- **上下文权威源单一**：商品上下文只有云端一个权威源，本地乐观拉取回写，避免多处数据不一致。
- **多租户能力预埋**：云端核心数据表第一天就带租户字段（当前写死默认租户，暂不启用隔离逻辑），为未来 SaaS 化多租户预留，避免事后改表的高昂代价。
- **图片存永久标识、展示时按需换签名**：上下文只存图片的永久存储标识，展示时才换成有时效的访问链接，保证上下文数据永不因链接过期而损坏。
- **反风控逻辑不重写**：真实 Cookie/UA/IP + 模拟人类输入 + 随机延迟 + 滑块人工介入，这套逻辑从成熟项目原样迁入，只做加法不做"优化"，确保上架稳定不触发平台风控。

## 6. 端到端数据流

一个商品从选品到上架的完整流转：

```
① 选品类 + 选主件               （本地前端）
        │  从 ERP 自动拉取产品白底图
        ▼
② 智能配件搭配 + 卖点提取        （云端 AI）
        │  规则库补配件；AI 从产品信息提取结构化卖点，写入商品上下文
        ▼
③ 布局规划 + 生成营销主图        （云端 AI，并行）
        │  卖点反哺 SKU 规划；按白底图生成多张主图（视角/风格/卖点差异化）
        ▼
④ 生成详情图 + SKU 配件图        （云端 AI + Canvas 合成）
        │  详情图与主图连续；SKU 图 AI 生主体、Java 确定性贴配件卡
        ▼
⑤ 成本计算 + 阶梯定价 + 标题     （云端）
        │  全部产物挂到商品上下文
        ▼
⑥ 人工在预览页选定方案、微调       （本地前端）
        ▼
⑦ 一键上架到电商平台             （本地 Playwright + 反风控）
           触发验证码/滑块 → 暂停提醒 → 人工划过 → 无缝续跑
```

全程数据都挂在**商品全局上下文**上，各环节读写同一份数据，不再有"手动搬运数据"的断点。

## 8. 竞争力与差异化

市面上多数工具只做其中一环——要么是 AI 生图工具，要么是电商上架/搬家工具。GOFU-AI 的差异化在于**把两环打通并互相增强**：

**① 防比价，不只是"生成好看的图"**
普通 AI 生图追求单张好看；GOFU-AI 追求**整组商品图的差异化**——每款的风格、构图、视角、画面卖点都不同，专门对抗电商平台的比价机制。同时用"AI 生主体 + 确定性合成配件"保证配件 100% 真实，规避虚假宣传风险。

**② 卖点反哺：生图与上架不是两个工具，是一条数据链**
生图阶段提取的商品卖点，直接驱动上架阶段的 SKU 规划与定价策略。这条"视觉流反哺结构流"的链路，是单一工具无法提供的——它需要一个贯穿全流程的商品上下文作为底座。

**③ 架构护栏，易维护、可扩展**
云/本地物理隔离、上下文单一权威源、多租户预埋、多模型可降级——这些设计让系统在扩展新品类、新平台、新模型时改动可控。当前从花洒扩展到厨房架类品，主要是新增品类的配置与提示词，核心链路不动，印证了架构的可扩展性。

**④ 真实环境反风控 + 人工兜底**
上架用真实浏览器环境操作，模拟人类输入节奏，遇验证码/滑块自动暂停并提醒人工，划过后无缝续跑——在"自动化效率"和"平台合规稳定"之间取得平衡。
